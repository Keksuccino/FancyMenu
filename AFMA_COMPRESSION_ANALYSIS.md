# AFMA Compression Analysis Notes

## Message 1: Best Wins

**Read**
AFMA is already a small video codec. The encoder is a greedy per-frame planner that tries `full`, `delta_rect`, `residual`, `sparse`, `copy_rect_*`, and `block_inter`, then picks the smallest candidate under a few hard thresholds in [AfmaEncodePlanner.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaEncodePlanner.java#L111) and [AfmaEncodePlanner.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaEncodePlanner.java#L175). Runtime decode is very decode-speed-oriented: a stream thread reads payloads, applies them directly into `NativeImage` memory, and only uploads dirty rects in [AfmaTexture.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L440), [AfmaTexture.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L675), and [AfmaNativeImageHelper.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaNativeImageHelper.java#L71). That means the best wins are the ones that move more work into encoding and metadata packing, not into per-frame decode math.

The two biggest current structural limits are that AFMA is still packaged as many ZIP entries plus JSON in [AfmaArchiveWriter.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaArchiveWriter.java#L45) and [AfmaDecoder.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaDecoder.java#L102), and the planner’s size model assumes per-payload deflate rather than global chunk compression in [AfmaPayloadMetricsHelper.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaPayloadMetricsHelper.java#L22).

**Best Wins**
- `1.` Replace ZIP-per-payload with a chunked solid container plus a binary frame index. Right now every payload pays entry overhead, JSON/path overhead, and loses cross-payload compression. This is likely the single biggest format-level size win, and decode can get faster because sequential playback no longer has to open and inflate lots of tiny ZIP entries.
- `2.` Add temporal frame collapsing, not just `SAME`. AFMA already supports variable frame durations via metadata, but today duplicate elision still emits one descriptor per source frame in [AfmaEncodePlanner.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaEncodePlanner.java#L148). A preprocessing pass that merges exact duplicate runs, and optionally near-duplicate runs under a perceptual threshold, would cut both payload count and index size while making playback cheaper.
- `3.` Make `block_inter` motion search much stronger without changing the decoder. The current search only tests vectors derived from global axis candidates in [AfmaRectCopyDetector.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaRectCopyDetector.java#L75) and [AfmaEncodePlanner.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaEncodePlanner.java#L763). Add per-tile local refinement around the best global vectors, and optionally a second long-term reference frame, so more tiles become `COPY` or `COPY_SPARSE` instead of `RAW`.
- `4.` Add perceptual BIN_INTRA candidates. Full and patch payloads are still exact pixels fed into the current BIN_INTRA modes in [AfmaBinIntraPayloadHelper.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaBinIntraPayloadHelper.java#L33). An encoder-only quantized candidate set, especially alpha-aware palette reduction for UI-like frames, can massively improve indexed payload hit rate while leaving decode unchanged.
- `5.` Make GOP/keyframe placement adaptive instead of mostly preset-driven. The presets only widen the search window and keyframe interval in [AfmaOptimizationPreset.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaOptimizationPreset.java#L12), and the planner still uses a fixed maximum interval in [AfmaEncodeOptions.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/creator/AfmaEncodeOptions.java#L21). For smallest-file mode, I would let GOPs grow until inter coding stops winning or perceptual drift rises, because AFMA playback is sequential and does not need random frame seek.
- `6.` Pack the frame index as binary opcodes with run-lengths. `frame_index.json` repeats field names, paths, and `"same"` descriptors in [AfmaDecoder.java](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaDecoder.java#L152). A binary index with payload IDs, varints, and SAME/repeat runs would reduce file size and speed up startup parsing.

**Order**
The best first implementation for fast, low-risk gains is temporal frame collapsing, because it can shrink files a lot without touching runtime decode at all. The biggest long-term win is the container redesign, but that should ship together with a new chunk-aware size estimator so the planner does not keep optimizing for ZIP semantics. After that, I’d do better `block_inter` search, because it is the most “video-codec-like” improvement you can add while keeping the current decoder hot path essentially unchanged.

I didn’t change code here. If you want the strongest next step, I’d start with `temporal frame collapsing + adaptive durations`; if you want the biggest total payoff, I’d start with `binary index + solid chunked container`.

## Message 2: Memory Consumption Safety

Mostly no, as long as we keep runtime as the hard constraint.

Current playback memory is basically:
- one full-frame `NativeImage` canvas
- one extra full-frame reference canvas only if the sequence uses `block_inter` in [AfmaTexture.java#L88](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L88) and [AfmaTexture.java#L861](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L861)
- up to 2 prefetched decoded/payload frames because `PREFETCH_QUEUE_SIZE` is `2` in [AfmaTexture.java#L34](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L34) and [AfmaTexture.java#L82](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L82)

With that baseline:

- Temporal frame collapsing is safe. It should reduce or keep runtime memory flat.
- Binary frame index is safe. It should reduce startup memory versus JSON parsing in [AfmaDecoder.java#L128](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaDecoder.java#L128) and [AfmaDecoder.java#L152](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaDecoder.java#L152).
- Adaptive GOP/keyframe placement is safe. It changes decisions, not the runtime memory model.
- Stronger `block_inter` search is safe if it stays single-reference. More `block_inter` frames do not create more than the one existing reference canvas.
- Perceptual/quantized `BIN_INTRA` candidates are runtime-safe. Decode still ends up as the same `int[]`/canvas writes in [AfmaTexture.java#L499](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L499).
- A chunked solid container is safe only if we keep chunk buffering tight. Today payloads are loaded per frame with `readAllBytes()` in [AfmaTexture.java#L499](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L499) and [AfmaTexture.java#L511](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaTexture.java#L511), so a badly chosen large chunk cache could be worse than the current model.

The one idea from my last message that would inherently worsen runtime memory is multi-reference motion compensation. If we add a second long-term reference frame, that costs another full-frame canvas. If “memory must not get worse” is strict, I would not do that.

If you also mean encoder peak RAM, then two things can grow it unless we deliberately trade time for memory:
- More `BIN_INTRA` candidate modes, especially with parallel candidate building in [AfmaBinIntraPayloadHelper.java#L37](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaBinIntraPayloadHelper.java#L37) and [AfmaBinIntraPayloadHelper.java#L50](/mnt/e/CODING/WORKSPACES/IntelliJ/Minecraft%20Mods/fancymenu-1.21.1/common/src/main/java/de/keksuccino/fancymenu/util/resource/resources/texture/afma/AfmaBinIntraPayloadHelper.java#L50)
- Stronger motion search if we parallelize or keep large temporary work buffers

So the safe rule set is:
- no extra runtime reference canvases
- no larger prefetch queue
- no large decompressed chunk cache
- extra encoder work should prefer sequential search and reusable scratch buffers over parallel RAM growth
